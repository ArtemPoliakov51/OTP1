# SPRINT 7 - PLAN
## 14.4.2026 - 28.4.2026

### SPRINT GOAL
The main goal of the sprint is to conduct functional and non-fuctional testing on the Attendance Checker application and system.
This includes writing and executing test cases, documenting and tracking issues and creating reports for the User Acceptance Testing and the Heuristic evaluation, 
as well as for all the acceptance testing done during this sprint. 
The plan is to also use Jenkins to generate the SonarQube statistic code review analysis for the project.

### BACKLOG FROM PREVIOUS SPRINT
The focus of the previous sprint was on database localization and QA of the code. 
Unfortunately, the team did not manage to do code clean-up and reafacoring to increase the code quality. 
So, two code quality issues should be resolved in this sprint: reducing duplicate code and cleaning up the code based on the issues found with CheckStyle.

### PLANNED TASKS
Here is the list of planned tasks for this sprint:

- Creating a test plan with test tasks for functional and non-functional testing.
- Cleanup the code and conduct final unit testing to ensure that the functionality is still working as intended.
- Create a table for tracking and addressing issues identified during testing.
- Use Jenkins to generate the static code analysis report with SonarQube when Project is build succefully.
- Make sure Docker image and deployment work correctly.
- Conduct heuristic evaluation for the software.
- Carry out a User Acceptance Test, based on the test tasks, and report results.
- Record specification and architecture changes based on the test that were carried out.

### TEAM CAPACITY & ASSUMPTIONS
Project has three members and the sprint is two weeks long. 
During those two weeks assuming that three days a week can be reserved for full-time project work, 
each team member can work about six hours a day so 18 hours a week and 36 hours total during the sprint. 
That makes the whole team capacity 108 hours for the whole sprint. 
These are just assumptions, the correct capacity calculation might differ based on the amount of tasks and other obligations outside the project.

### DEFINITION OF DONE
Sprint is done when the deadline is reached and all the tasks are finished. 
All the selected product backlog items need to be implemented for the sprint to be done.
